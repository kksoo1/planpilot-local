# PlanPilot Local 오류/로딩/빈 상태 정책

이 문서는 JSON 내보내기 기능을 구현하기 전에 PlanPilot Local의 로딩, 오류, 빈 상태 UX를 어디까지 보강할지 정리한다.

현재 문서는 정책 문서이며, 실제 오류/로딩 UI, JSON 내보내기, JSON 가져오기 기능은 아직 구현하지 않는다.

## 목표

- IndexedDB 기반 로컬 앱에서 사용자가 현재 상태를 이해할 수 있게 한다.
- 오류가 발생했을 때 데이터 손상이나 중복 동작을 만들지 않는다.
- 서버 API, `localStorage`, 로그인, 클라우드 동기화 없이 로컬 상태만 다룬다.
- JSON 내보내기/가져오기 전에 공통 상태 표시 기준을 먼저 정한다.

## 현재 상태

- 앱 초기화는 `initializeApp()` 흐름으로 수행된다.
- 현재 전용 초기 로딩 화면이나 초기화 오류 화면은 없다.
- rule-based 추천 업무는 로컬 계산으로 생성되며, 전용 로딩/오류 UI는 없다.
- 업무/프로젝트 CRUD는 store action을 통해 IndexedDB에 반영된다.
- 일부 화면에는 목록이 비어 있을 때 표시할 문구가 있으나, 앱 전체 공통 empty/error/loading 정책은 아직 없다.
- 백업/내보내기 기능은 정책만 문서화되어 있고 아직 구현하지 않았다.

## 로딩 상태가 필요한 영역

1차 보강 후보:

- 앱 초기 데이터 로딩
- IndexedDB 초기화와 기본 데이터 생성
- rule-based 추천 업무 생성
- 향후 JSON 내보내기 파일 생성
- 향후 JSON 가져오기 파일 읽기와 검증

정책:

- 로딩 상태는 사용자가 중복 클릭하거나 데이터가 사라진 것으로 오해하지 않도록 짧고 명확하게 표시한다.
- 로컬 계산이 아주 짧게 끝나는 영역은 과한 spinner보다 상태 문구 또는 비활성화 기준을 우선 검토한다.
- JSON 내보내기/가져오기는 파일 처리 시간이 짧더라도 실패 가능성이 있으므로 별도 진행 상태를 둘 수 있다.

## 오류 상태가 필요한 영역

1차 보강 후보:

- IndexedDB 읽기 실패
- IndexedDB 쓰기 실패
- 업무 추가/수정/삭제 실패
- 프로젝트 추가/수정/삭제 실패
- 추천 업무 생성 실패
- 향후 백업 파일 생성 실패
- 향후 가져오기 파일 파싱 실패
- 향후 가져오기 데이터 검증 실패

정책:

- 오류 문구는 사용자가 다음 행동을 알 수 있게 작성한다.
- IndexedDB 쓰기 실패 시 성공한 것처럼 보이게 하지 않는다.
- 가져오기 실패 시 기존 데이터는 유지되어야 한다.
- 추천 업무 생성 실패는 업무/프로젝트 CRUD를 막지 않는 보조 오류로 취급한다.
- 서버 전송, 원격 로그, 외부 오류 수집은 추가하지 않는다.

## 빈 상태가 필요한 영역

1차 보강 후보:

- 업무 없음
- 프로젝트 없음 또는 기본 프로젝트만 있음
- 검색 결과 없음
- 필터 결과 없음
- 추천 업무 없음
- 지난 마감 업무 없음
- 7일 이내 마감 업무 없음

정책:

- 빈 상태는 오류가 아니라 정상 상태로 표시한다.
- 검색 결과 없음과 업무 없음은 구분한다.
- 필터 결과 없음은 필터 해제 또는 조건 변경을 유도할 수 있다.
- 추천 업무 없음은 rule-based 추천 기준에 맞는 미완료 업무가 없다는 의미로 다룬다.
- 기본 프로젝트만 있는 상태는 프로젝트 데이터 손실이 아니라 초기 상태일 수 있다.

## MVP 이후 우선순위

1. 오류/로딩/빈 상태 정책 문서화
2. 공통 empty/error/loading 표시 기준 정리
3. 기존 화면의 빈 상태 문구와 조건 점검
4. JSON 내보내기 전용 기능 구현
5. JSON 내보내기 오류 처리와 수동 테스트
6. JSON 가져오기/복원은 별도 단계에서 정책을 확정한 뒤 구현

## 구현 전 중단 조건

- `App.css` 대규모 수정이 필요하다.
- DB schema 변경이 필요하다.
- store API 구조 변경이 필요하다.
- 기존 수동 테스트 범위가 너무 커진다.
- 서버 API, `localStorage`, 로그인, 클라우드 동기화, 알림 권한 요청, Capacitor가 필요해진다.
- JSON export/import 구현 범위가 동시에 커져 기존 데이터 손상 위험을 판단하기 어렵다.

## 수동 테스트 항목 후보

구현 후 확인할 항목:

- 앱 초기 로딩 중 화면이 깨지지 않는다.
- IndexedDB 초기화 후 기본 프로젝트가 유지된다.
- 업무가 없을 때 빈 상태가 오류처럼 보이지 않는다.
- 검색 결과가 없을 때 검색 결과 없음 상태가 표시된다.
- 필터 결과가 없을 때 필터 결과 없음 상태가 표시된다.
- 추천 업무가 없을 때 추천 업무 없음 상태가 표시된다.
- 지난 마감 업무가 없을 때 빈 상태가 표시된다.
- 7일 이내 마감 업무가 없을 때 빈 상태가 표시된다.
- DB 오류는 일반 수동 재현이 어려우므로 정책 문서상 보류하거나 별도 개발용 재현 절차를 마련한다.
- 백업/내보내기 오류는 JSON 내보내기 기능 구현 후 테스트한다.
- 가져오기 오류는 JSON 가져오기 기능 구현 후 테스트한다.

## 금지 범위

- 이 정책 문서 작성 단계에서는 src 코드를 수정하지 않는다.
- JSON 내보내기/가져오기 기능을 구현하지 않는다.
- DB schema를 변경하지 않는다.
- 서버 API, `localStorage`, 로그인, 클라우드 동기화, 알림 권한 요청, Capacitor를 추가하지 않는다.

## 다음 코드 작업의 최소 적용 범위

다음 코드 작업에서 바로 적용할 범위는 전체 오류/로딩 체계가 아니라 기존 화면의 빈 상태 보강으로 제한한다.

1차 적용 후보:

- 업무 없음
- 검색 결과 없음
- 필터 결과 없음
- 추천 업무 없음
- 지난 마감 업무 없음
- 7일 이내 마감 업무 없음

이 범위는 DB schema, store API, 서버 API, `App.css` 대규모 변경 없이 확인 가능한 영역이다. 구현 시에도 새 기능보다 현재 화면의 빈 목록이 사용자에게 정상 상태로 이해되는지 확인하는 데 집중한다.

## 다음 코드 작업에서 보류할 항목

다음 항목은 JSON 내보내기 구현 전 최소 empty 상태 보강 범위에 포함하지 않는다.

- IndexedDB 오류 UI
- CRUD 실패 error boundary
- JSON export 실패 UI
- JSON import 실패 UI
- 전역 loading overlay

보류 이유:

- IndexedDB와 CRUD 실패는 수동 재현과 검증 범위가 크다.
- JSON export/import 오류 UI는 해당 기능이 구현된 뒤 전용 흐름에서 다루는 편이 안전하다.
- 전역 loading overlay는 앱 전체 레이아웃과 상태 흐름에 영향을 줄 수 있으므로 MVP 이후 첫 보강 범위로는 크다.

## JSON export 전 필수 여부

JSON export 기능은 단독 버튼과 파일 생성 흐름으로 구현할 수 있으므로, 전체 앱 공통 error/loading 체계를 먼저 완성해야만 구현할 수 있는 기능은 아니다.

권장 기준:

- JSON export 전에는 기존 화면의 빈 상태 문구를 먼저 점검/보강한다.
- JSON export 구현 시에는 공통 전역 체계보다 export 전용 성공/실패 메시지를 먼저 고려한다.
- JSON import는 데이터 손상 위험이 있으므로 export 이후 별도 단계에서 오류/검증 UI 정책을 확정한다.

## 다음 구현 우선순위 제안

1. 현재 화면의 빈 상태 문구 점검/보강
2. JSON export 전용 성공/실패 메시지 정책 정리
3. JSON export 구현
4. import 전 오류/검증 UI 정책 정리

## JSON export 전용 피드백 정책

JSON export는 전체 앱 공통 error/loading 체계보다 export 전용 성공/실패 메시지를 먼저 적용한다.

- 성공 메시지는 SettingsView 내부의 export 버튼 근처에 짧게 표시하는 방향을 1차 후보로 둔다.
- 실패 메시지도 같은 영역에 표시해 사용자가 다시 시도할 수 있게 한다.
- `alert`는 보조 후보이며, 전역 toast와 전역 loading overlay는 아직 도입하지 않는다.
- export 중에는 버튼 비활성화 또는 짧은 진행 문구를 우선 검토한다.
- 실패해도 기존 IndexedDB 데이터는 변경하지 않는 기준을 유지한다.

## JSON export 실패 경로 보류 기준

JSON export 실패 경로는 현재 일반 브라우저 수동 테스트에서 안정적으로 재현하기 어렵다.

- 데이터 직렬화 실패, Blob 생성 실패, object URL 생성 실패, 다운로드 링크 클릭 실패, 브라우저 다운로드 차단, 알 수 없는 예외를 실패 후보로 둔다.
- 현재 수동 테스트 문서에는 실패 경로를 보류/확인 필요로 기록한다.
- 실패 메시지는 코드상 `try/catch` 경로로 표시 가능해야 하지만, 실제 표시 확인은 실패 주입 방법이 마련된 뒤 수행한다.
- 실패 경로 확인을 위해 src 코드를 임시로 변경하지 않는다.
- 단위 테스트, download helper 분리, 테스트 hook 도입은 별도 정책 작업에서 결정한다.
- JSON import/복원 오류 UI는 아직 구현하지 않고 별도 단계로 유지한다.

## JSON import/복원 오류 보류 기준

JSON import/복원은 아직 구현하지 않는다. 오류 UI도 실제 import 기능보다 먼저 구현하지 않는다.

- 1차 import 후보는 실제 DB 반영이 아니라 파일 검증 또는 미리보기다.
- 잘못된 JSON, `format` 불일치, 미지원 `schemaVersion`, 필수 필드 누락, 깨진 `projectId` 참조를 오류 후보로 둔다.
- 검증 실패 또는 사용자의 취소 시 기존 IndexedDB 데이터는 변경하지 않는 기준을 유지한다.
- 실제 복원 오류와 rollback UI는 복원 방식이 전체 덮어쓰기인지 병합인지 확정한 뒤 설계한다.
- 서버 API, `localStorage`, 로그인, 클라우드 동기화, 알림 권한 요청, Capacitor를 오류 보고나 복원 흐름에 추가하지 않는다.

## 최소 범위 수동 테스트

다음 코드 작업 후에는 아래 항목을 우선 확인한다.

- 업무가 없을 때 전체 업무 화면이 정상 빈 상태로 보인다.
- 검색 결과가 없을 때 검색 결과 없음 상태가 업무 없음 상태와 구분된다.
- 프로젝트 필터 결과가 없을 때 필터 결과 없음 상태가 표시된다.
- 추천 업무가 없을 때 추천 업무 없음 상태가 표시된다.
- 지난 마감 업무가 없을 때 빈 상태가 표시된다.
- 7일 이내 마감 업무가 없을 때 빈 상태가 표시된다.
- JSON export 구현 후에는 export 성공/실패 메시지를 별도로 확인한다.
