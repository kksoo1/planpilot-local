# PlanPilot Local 데이터 백업/내보내기 정책

이 문서는 PlanPilot Local의 IndexedDB 로컬 데이터를 백업하고 내보내기 전에 따라야 할 정책을 정리한다.

현재 문서는 정책 문서이며, 실제 백업/내보내기/가져오기 기능은 아직 구현하지 않는다.

## 목표

- 사용자가 로컬에 저장한 업무, 프로젝트, 설정 데이터를 사용자가 직접 보관할 수 있게 한다.
- 개인정보가 서버로 전송되지 않는 privacy-first 원칙을 유지한다.
- `localStorage`, 로그인, 클라우드 동기화 없이 IndexedDB 데이터를 다룬다.
- 가져오기/복원은 데이터 손상 위험이 있으므로 내보내기보다 나중 단계로 분리한다.

## 백업/내보내기 대상 데이터

내보내기 1차 대상:

- `tasks`
- `projects`
- `appSettings`

각 테이블은 현재 Dexie schema version 1 기준으로 내보낸다.

`tasks`에는 업무 제목, 메모, 마감일, 우선순위, 상태, 프로젝트 연결, 태그, 예상 시간, 생성/수정/완료 시각, 정렬 순서를 포함한다.

`projects`에는 프로젝트 이름, 설명, 색상, 생성/수정/보관 시각을 포함한다.

`appSettings`에는 현재 설정 값인 `theme`, `language`, `aiProvider`, `enableNotifications`, `firstLaunchCompleted`, `createdAt`, `updatedAt`을 포함하는 방향을 기본 후보로 둔다.

## 내보내기 형식 후보

1차 후보는 사람이 읽기 쉬운 JSON이다.

권장 JSON 구조 후보:

```json
{
  "format": "planpilot-local-backup",
  "version": 1,
  "exportedAt": "ISO-8601 string",
  "source": {
    "app": "PlanPilot Local",
    "storage": "Dexie + IndexedDB"
  },
  "data": {
    "tasks": [],
    "projects": [],
    "appSettings": null
  }
}
```

정책:

- JSON은 pretty print 형식으로 내보낸다.
- `format`, `version`, `exportedAt` 같은 메타 정보를 포함한다.
- 추후 CSV 내보내기는 업무 목록 중심의 보조 기능으로 검토한다.
- CSV는 설정과 프로젝트 참조 관계를 온전히 보존하기 어렵기 때문에 복원용 기본 형식으로 사용하지 않는다.

## 가져오기/복원 정책 후보

가져오기는 데이터 손상 위험이 있으므로 내보내기 이후 별도 단계에서 설계한다.

후보 정책:

- 전체 덮어쓰기
  - 기존 `tasks`, `projects`, `appSettings`를 백업 파일 기준으로 교체한다.
  - 구현 전 사용자 확인 문구와 기존 데이터 보존 방식을 먼저 정해야 한다.
- 병합
  - 기존 데이터와 백업 데이터를 합친다.
  - 중복 ID 처리, 충돌 시 우선순위, 프로젝트 참조 무결성 검증이 필요하다.
- 기존 데이터 백업 후 복원
  - 가져오기 전에 현재 데이터를 별도 JSON으로 내보낼 수 있어야 한다.
  - 복원 실패 시 기존 데이터를 유지하거나 되돌리는 기준이 필요하다.

1차 구현 권장 순서:

1. 내보내기 전용 기능을 먼저 구현한다.
2. 내보낸 JSON 구조와 수동 테스트 기준을 안정화한다.
3. 가져오기/복원은 전체 덮어쓰기와 병합 중 하나를 별도 정책으로 확정한 뒤 구현한다.

## 중복 ID 처리 기준

가져오기 구현 전까지는 중복 ID 처리를 확정하지 않는다.

후보:

- 동일 ID가 있으면 가져오기 중단
- 동일 ID가 있으면 백업 파일 값으로 덮어쓰기
- 동일 ID가 있으면 새 ID를 생성해 병합

현재 권장안은 안전을 위해 동일 ID 충돌 시 가져오기를 중단하고 사용자에게 원인을 표시하는 것이다.

## appSettings 처리 기준

`appSettings`는 백업에 포함하는 방향을 기본 후보로 둔다.

포함 이유:

- 설정 화면은 현재 읽기 전용이지만 앱 상태의 일부다.
- `firstLaunchCompleted` 같은 내부 상태도 로컬 사용 경험에 영향을 줄 수 있다.

주의점:

- `createdAt`은 백업 시점에 새로 만들지 않고 원본 값을 유지하는 후보가 안전하다.
- `updatedAt`은 복원 방식에 따라 원본 유지 또는 복원 시각 갱신 중 하나를 정해야 한다.
- 1차 내보내기는 원본 `createdAt`, `updatedAt`을 그대로 보존한다.
- 가져오기 구현 시 설정을 포함할지, 제외할지, 선택 가능하게 할지 별도 확인한다.

## 데이터 무결성 검증 기준

가져오기/복원 전에는 다음 검증이 필요하다.

- 최상위 `format` 값이 `planpilot-local-backup`인지 확인한다.
- 지원 가능한 `version`인지 확인한다.
- `tasks`, `projects`, `appSettings` 구조가 예상 타입과 맞는지 확인한다.
- 각 레코드의 필수 필드를 확인한다.
  - task: `id`, `title`, `priority`, `status`, `projectId`, `tags`, `createdAt`, `updatedAt`
  - project: `id`, `name`, `createdAt`, `updatedAt`
  - appSettings: `theme`, `language`, `aiProvider`, `enableNotifications`, `firstLaunchCompleted`, `createdAt`, `updatedAt`
- 날짜 문자열은 ISO-8601 문자열 후보로 검증한다.
- task의 `projectId`가 존재하는 project를 참조하는지 확인한다.
- `default` 프로젝트가 없으면 가져오기 중단 또는 기본 프로젝트 보정 정책을 먼저 결정한다.
- 알 수 없는 필드는 1차 구현에서 보존하지 않고 무시하는 방향을 후보로 둔다.
- 알 수 없는 필드를 무시할 경우 사용자에게 호환성 주의 문구를 제공한다.

## 보안/프라이버시 기준

- 백업/내보내기는 로컬 브라우저 안에서만 처리한다.
- 백업 파일은 사용자가 직접 다운로드하거나 저장한다.
- 서버 API를 추가하지 않는다.
- 외부 서버로 데이터를 전송하지 않는다.
- `localStorage`를 사용하지 않는다.
- 로그인이나 계정 기능을 추가하지 않는다.
- 클라우드 동기화를 추가하지 않는다.
- 외부 AI API로 백업 데이터를 보내지 않는다.

## MVP 이후 우선순위

1차 기능 고도화 후보:

1. JSON 내보내기 전용 기능
2. 내보낸 JSON 구조 수동 테스트
3. 가져오기/복원 정책 확정
4. 가져오기 전 현재 데이터 백업 기능
5. 가져오기/복원 구현

CSV 내보내기는 JSON 내보내기가 안정화된 뒤 보조 기능으로 검토한다.

## 수동 테스트 항목 후보

내보내기:

- 내보낸 JSON에 `format`, `version`, `exportedAt`, `data`가 포함된다.
- 내보낸 JSON에 `tasks`, `projects`, `appSettings`가 포함된다.
- 업무/프로젝트/설정 데이터 수가 화면과 일치한다.
- 내보내기 후 기존 IndexedDB 데이터가 변경되지 않는다.
- 새로고침 후 기존 데이터가 유지된다.
- 서버 API, `localStorage`, 로그인, 클라우드 동기화가 추가되지 않는다.

가져오기/복원:

- 잘못된 JSON 파일을 선택하면 기존 데이터가 유지된다.
- 지원하지 않는 `format` 또는 `version`이면 가져오기를 중단한다.
- 필수 필드가 빠진 파일은 가져오기를 중단한다.
- 존재하지 않는 프로젝트를 참조하는 task가 있으면 가져오기를 중단하거나 명시된 보정 정책을 따른다.
- 가져오기 취소 시 기존 데이터가 유지된다.
- 가져오기 실패 시 기존 데이터가 유지된다.
- 가져오기 성공 후 새로고침해도 데이터가 유지된다.

## 구현 전 중단 조건

- DB schema 변경이 필요하다고 판단되는 경우
- 기존 IndexedDB migration이 필요한 경우
- 파일 파싱 오류 처리 정책이 정해지지 않은 경우
- 가져오기 실패 시 기존 데이터 보존 방식이 정해지지 않은 경우
- 중복 ID 처리 정책이 정해지지 않은 경우
- 프로젝트 참조 무결성 보정 방식이 정해지지 않은 경우
- `App.css` 대규모 수정이 필요한 경우
- 서버 API, `localStorage`, 로그인, 클라우드 동기화, 알림 권한 요청, Capacitor가 필요해지는 경우
